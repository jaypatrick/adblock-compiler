## Overview

This issue tracks the full testing story for `adblock-compiler` across three dimensions:

1. **Functional integration testing** — What's in place, what's missing, and how to build it out using the existing Cloudflare infrastructure (including Browser Rendering).
2. **Unit test coverage** — Current state, gaps, and a roadmap to comprehensive coverage.
3. **Test infrastructure consolidation** — How to unify the test stack, reduce duplication, and leverage best-of-breed frameworks.

---

## 1. Functional Integration Testing

### Current State

- `worker/handlers/browser.ts` and `src/platform/BrowserFetcher.ts` have structural interfaces (`IPlaywrightBrowser`, `IPlaywrightPage`) defined for testability, but actual integration tests exercising the **live Cloudflare Browser Rendering binding** do not appear to exist.
- Issue [#983](https://github.com/jaypatrick/adblock-compiler/issues/983) calls for integration tests but leaves the approach open.
- Issue [#984](https://github.com/jaypatrick/adblock-compiler/issues/984) identifies Browser Rendering use cases (stale detection, `useBrowser` auto-detection, diff screenshots) that are integration-testable scenarios with no corresponding tests yet.
- The `WorkerCompilerDependencies` injection surface on `WorkerCompiler` is well-designed for integration testing (swap `browserConnector`, `fetcher`, etc.) but is unused in any test suite.
- No evidence of end-to-end tests against the **Worker API** (e.g., `POST /compile`, `POST /api/browser/monitor`, SSE streaming).

### Gaps

- [ ] No integration tests for `BrowserFetcher` against a real or mocked Browser Rendering binding.
- [ ] No Worker-level HTTP integration tests (request → handler → response).
- [ ] No SSE streaming integration tests (`POST /compile` with `stream: true`).
- [ ] No queue round-trip tests (`POST /compile` → `useQueue: true` → polling or callback).
- [ ] No multi-source compilation integration test (fetch N sources → compile → assert rule count/output).
- [ ] No `useBrowser: true` source integration test (Browser Rendering → filter list extraction → compile).

### Recommendations

#### Use `wrangler dev` + Deno `fetch` for Worker HTTP Integration Tests

Run the Worker locally with `wrangler dev` and write Deno test scripts that hit its HTTP endpoints directly:

```typescript
// tests/integration/worker/compile.test.ts
Deno.test("POST /compile returns compiled rules", async () => {
  const response = await fetch("http://localhost:8787/compile", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      name: "test-list",
      sources: [{ url: "https://raw.githubusercontent.com/.../test-hosts.txt" }],
    }),
  });
  const result = await response.json();
  assert(result.rules.length > 0);
});
```

#### Use `@cloudflare/playwright` Mock for BrowserFetcher Integration Tests

Leverage the `BrowserConnector` / `IPlaywrightBrowser` interfaces already defined in `src/platform/BrowserFetcher.ts`. These were explicitly designed for test injection:

```typescript
// tests/integration/platform/BrowserFetcher.test.ts
import { BrowserFetcher } from "../../src/platform/BrowserFetcher.ts";

const mockBrowser: IPlaywrightBrowser = {
  newPage: async () => ({
    goto: async () => {},
    content: async () => "<html><body><pre>||ads.example.com^\n||tracker.io^</pre></body></html>",
    evaluate: async () => "||ads.example.com^\n||tracker.io^",
    close: async () => {},
  }),
  close: async () => {},
};

Deno.test("BrowserFetcher extracts filter rules from JS-rendered page", async () => {
  const fetcher = new BrowserFetcher({ fetch } as IBrowserWorker, {}, async () => mockBrowser);
  const content = await fetcher.fetch("https://example.com/filters.txt");
  assertStringIncludes(content, "||ads.example.com^");
});
```

#### Consider Cloudflare Vitest Integration for Worker-native Tests

Use `@cloudflare/vitest-pool-workers` to run tests in the Workers runtime (access to KV, D1, R2, Queue, Browser bindings via `env`):

```typescript
// vitest.config.ts
import { defineWorkersConfig } from "@cloudflare/vitest-pool-workers/config";

export default defineWorkersConfig({
  test: {
    poolOptions: {
      workers: {
        wrangler: { configPath: "./wrangler.toml" },
      },
    },
  },
});
```

This gives access to real binding stubs (`env.COMPILATION_CACHE`, `env.METRICS`, `env.DB`) without deploying.

## 2. Unit Test Coverage

### Current State

The codebase has clear testability-by-design:
- `WorkerCompilerDependencies` makes `WorkerCompiler` fully injectable (validator, pipeline, fetcher, browserConnector, etc.).
- `BrowserFetcher`, `HttpFetcher`, `CompositeFetcher`, `PreFetchedContentFetcher` all implement `IContentFetcher` and can be swapped.
- `withRetry` / `RetryStrategies` in `src/utils/AsyncRetry.ts` are pure functions — trivially unit-testable.
- `ConfigurationValidator`, `TransformationPipeline`, `HeaderGenerator` all appear to be injectable/instantiable directly.

However, based on code search results, formal unit test files (`.test.ts`) are not surfaced in significant numbers. The `tests/` directory or co-located `*.test.ts` files appear sparse for the volume of logic in `src/`.

### Identified Gaps by Module

| Module | Testable? | Test Coverage Status |
|---|---|---|
| `src/utils/AsyncRetry.ts` (`withRetry`, `RetryStrategies`) | ✅ Pure functions | ❌ No tests found |
| `src/platform/BrowserFetcher.ts` | ✅ Injectable interfaces | ❌ No tests found |
| `src/platform/HttpFetcher.ts` | ✅ `IContentFetcher` impl | ❌ No tests found |
| `src/platform/CompositeFetcher.ts` | ✅ Composable | ❌ No tests found |
| `src/platform/WorkerCompiler.ts` | ✅ Full DI via `WorkerCompilerDependencies` | ❌ No tests found |
| `src/configuration/schemas.ts` (Zod schemas) | ✅ Pure validation | ❌ No tests found |
| `src/diagnostics/` (TracingContext, DiagnosticContext) | ✅ Pure | ❌ Partial (usage in examples only) |
| `src/utils/ErrorReporter.ts` | ✅ Injectable reporters | ❌ No tests found |
| `worker/handlers/queue.ts` (`processCompileMessage`, `getQueueStats`) | ⚠️ Needs env mock | ❌ No tests found |
| `worker/services/sentry-init.ts` (`withSentryWorker`) | ✅ HOF wrapping | ❌ No tests found |

### Recommended Unit Tests to Write First (Priority Order)

1. **`withRetry` and `RetryStrategies`** — Pure functions, high ROI, no mocking needed.
2. **Zod schema validation** (`ConfigurationSchema`, `ValidationReportSchema`, etc.) — Test valid + invalid inputs against every schema.
3. **`BrowserFetcher`** — Use `IPlaywrightBrowser` / `IPlaywrightPage` mocks (already defined).
4. **`CompositeFetcher` / `PreFetchedContentFetcher`** — Simple behavioral tests.
5. **`WorkerCompiler.compile()`** — Inject a `PreFetchedContentFetcher` + mock pipeline to test end-to-end compile without network.
6. **`withSentryWorker`** — Test DSN-absent passthrough and DSN-present wrapping behavior.
7. **`createWorkerErrorReporter`** — Test each `reporterType` branch with mock env.

## 3. Test Infrastructure Consolidation

### Current Situation

- The project uses **Deno** as its runtime (`deno.json` / `deno task test`).
- Deno has a built-in test runner (`Deno.test`) and assertion library (`@std/assert`).
- The `@cloudflare/playwright` structural interfaces are shimmed in `cloudflare-workers-shim.ts` for Deno compatibility.

### Consolidation Ideas

#### A. Adopt `@cloudflare/vitest-pool-workers` for Worker-layer Tests

For anything that touches Cloudflare bindings (`env.KV`, `env.DB`, `env.BROWSER`, etc.), **Vitest with `cloudflare/vitest-pool-workers`** runs tests in the actual Workers runtime. This eliminates the need to manually mock KV, D1, R2, and Queue APIs.

- Works alongside Deno: use Deno for `src/` pure unit tests, Vitest for `worker/` integration/binding tests.
- Supports `wrangler.toml` binding stubs automatically.

#### B. Introduce a Shared Test Fixtures / Factories Module

Create `tests/fixtures/` with reusable:
- Sample filter list content (hosts, adblock, RPZ formats)
- Mock `IContentFetcher` implementations
- Mock `IPlaywrightBrowser` / `IPlaywrightPage`
- Factory functions for `WorkerCompilerOptions` with sensible test defaults
- Sample `CompilerConfiguration` objects

#### C. Use Cloudflare Browser Rendering for Functional UI/E2E Tests

Leverage the existing `BROWSER` binding to write end-to-end tests that:
- Load the Angular frontend via Playwright
- Submit a compilation job via the UI
- Assert the result is rendered correctly

This is high-value for regression testing and directly uses infrastructure already wired up.

```typescript
// tests/e2e/compile-flow.test.ts (vitest-pool-workers)
import { launch } from "@cloudflare/playwright";

test("UI compile flow produces downloadable result", async ({ env }) => {
  const browser = await launch(env.BROWSER);
  const page = await browser.newPage();
  await page.goto("https://adblock-compiler.pages.dev/compile");
  // ... interact with UI, assert result card appears
  await browser.close();
});
```

#### D. AGTree AST Roundtrip / Fidelity Tests

Per issue [#993](https://github.com/jaypatrick/adblock-compiler/issues/993), **roundtrip fidelity tests** are called for (A→B→A translation). These should be systematically structured as parameterized unit tests across all format pairs, using AGTree as the AST IR:

```typescript
for (const [sourceFormat, targetFormat] of formatPairs) {
  Deno.test(`Roundtrip fidelity: ${sourceFormat} → ${targetFormat} → ${sourceFormat}`, () => {
    const original = loadSampleRules(sourceFormat);
    const translated = translate(original, sourceFormat, targetFormat);
    const roundtripped = translate(translated, targetFormat, sourceFormat);
    assertFidelity(original, roundtripped, threshold: 0.95);
  });
}
```

#### E. CI Integration (Related: [#986](https://github.com/jaypatrick/adblock-compiler/issues/986))

- Run `deno test` for `src/` unit tests on every PR.
- Run `vitest` (Cloudflare pool) for `worker/` tests on every PR.
- Run E2E tests (Browser Rendering) on deploy to staging only (not every PR — too slow).
- Publish `deno bench` results as CI artifacts per issue #986.
