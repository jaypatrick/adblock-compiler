# Issue Title: Testing Strategy: Functional Integration Tests, Unit Test Coverage, and Infrastructure Consolidation

## Overview

This issue tracks the full testing story for `adblock-compiler` across three dimensions:

1. **Functional integration testing** — What's in place, what's missing, and how to build it out using existing Cloudflare infrastructure (including Browser Rendering).
2. **Unit test coverage** — Current state, gaps, and a roadmap to comprehensive coverage.
3. **Test infrastructure consolidation** — How to unify the test stack, reduce duplication, and leverage best-of-breed frameworks.

---

## 1. Functional Integration Testing

### Current State

- `src/platform/BrowserFetcher.ts` defines `IPlaywrightBrowser` and `IPlaywrightPage` structural interfaces explicitly for test injection, but no actual integration tests exercise the Cloudflare Browser Rendering binding.
- Issue #983 calls for integration tests but leaves the approach open; issue #984 identifies specific Browser Rendering use cases (stale detection, `useBrowser` auto-detection, diff screenshots) with no corresponding tests.
- `WorkerCompilerDependencies` on `WorkerCompiler` provides full DI (swap `browserConnector`, `fetcher`, `validator`, `pipeline`, etc.) but no test suite uses it.
- No evidence of end-to-end tests against the Worker API (`POST /compile`, `POST /api/browser/monitor`, SSE streaming, queue round-trips).

### Gaps

- [ ] No integration tests for `BrowserFetcher` against a mocked or real Browser Rendering binding
- [ ] No Worker-level HTTP integration tests (request → handler → response)
- [ ] No SSE streaming integration tests (`POST /compile` with `stream: true`)
- [ ] No queue round-trip tests (`POST /compile` → `useQueue: true` → callback/poll)
- [ ] No multi-source compilation integration test
- [ ] No `useBrowser: true` source integration test

### Recommendations

**A. `wrangler dev` + Deno `fetch` for Worker HTTP integration tests**

```typescript
// tests/integration/worker/compile.test.ts
Deno.test("POST /compile returns compiled rules", async () => {
  const response = await fetch("http://localhost:8787/compile", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      name: "test-list",
      sources: [{ url: "https://raw.githubusercontent.com/.../test-hosts.txt" }],
    }),
  });
  const result = await response.json();
  assert(result.rules.length > 0);
});
```

**B. `IPlaywrightBrowser` structural mock for BrowserFetcher tests**

The interfaces in `src/platform/BrowserFetcher.ts` were explicitly designed for this:

```typescript
const mockBrowser: IPlaywrightBrowser = {
  newPage: async () => ({
    goto: async () => {},
    content: async () => "<pre>||ads.example.com^</pre>",
    evaluate: async () => "||ads.example.com^",
    close: async () => {},
  }),
  close: async () => {},
};

Deno.test("BrowserFetcher extracts filter rules from JS-rendered page", async () => {
  const fetcher = new BrowserFetcher({ fetch } as IBrowserWorker, {}, async () => mockBrowser);
  const content = await fetcher.fetch("https://example.com/filters.txt");
  assertStringIncludes(content, "||ads.example.com^");
});
```

**C. `@cloudflare/vitest-pool-workers` for Worker-native binding tests**

```typescript
// vitest.config.ts
import { defineWorkersConfig } from "@cloudflare/vitest-pool-workers/config";
export default defineWorkersConfig({
  test: {
    poolOptions: { workers: { wrangler: { configPath: "./wrangler.toml" } } },
  },
});
```

Gives access to real KV/D1/R2/Queue/Browser binding stubs without deploying. **wrangler 4.73** `persist: false` (from #1089) eliminates manual binding resets between test runs.

**D. Cloudflare Browser Rendering for E2E tests**

`@cloudflare/playwright` 1.1.2 is now stable. Use it for full Angular frontend E2E. Consider [Stagehand](https://github.com/browserbase/stagehand) (AI-powered browser automation, bundled with `@cloudflare/playwright` 1.0+) for higher-level E2E assertions.

## 2. Unit Test Coverage

### Design Is Testability-Forward — Coverage Is Lacking

The codebase has strong DI and pure-function design, but formal `.test.ts` files are sparse for the volume of production logic in `src/`.

### Gap Table

| Module | Testable | Gap |
|---|---|---|
| `src/utils/AsyncRetry.ts` (`withRetry`, `RetryStrategies`) | ✅ Pure functions | ❌ No tests |
| `src/platform/BrowserFetcher.ts` | ✅ Injectable interfaces | ❌ No tests |
| `src/platform/HttpFetcher.ts` | ✅ `IContentFetcher` impl | ❌ No tests |
| `src/platform/CompositeFetcher.ts` | ✅ Composable | ❌ No tests |
| `src/platform/WorkerCompiler.ts` | ✅ Full DI via `WorkerCompilerDependencies` | ❌ No tests |
| `src/configuration/schemas.ts` (Zod) | ✅ Pure validation | ❌ No tests |
| `src/diagnostics/` (TracingContext) | ✅ Pure | ❌ Partial (examples only) |
| `src/utils/ErrorReporter.ts` | ✅ Injectable reporters | ❌ No tests |
| `worker/handlers/queue.ts` | ⚠️ Needs env mock | ❌ No tests |
| `worker/services/sentry-init.ts` (`withSentryWorker`) | ✅ HOF | ❌ No tests |

### Priority Order for First Tests

1. **`withRetry` and `RetryStrategies`** — Pure functions, zero mocking, highest ROI
2. **All Zod schemas** — Parameterized valid + invalid inputs for every schema in `src/configuration/schemas.ts` and `worker/schemas.ts`
3. **`BrowserFetcher`** — Structural mocks already defined in source
4. **`CompositeFetcher` / `PreFetchedContentFetcher`** — Simple behavioral tests
5. **`WorkerCompiler.compile()`** — Inject `PreFetchedContentFetcher` + mock pipeline, no network required
6. **`withSentryWorker`** — DSN-absent passthrough + DSN-present wrapping
7. **`createWorkerErrorReporter`** — Each `reporterType` branch with mock `Env`
8. **Clerk Zod schemas** (`ClerkWebhookEventSchema`, `ClerkJWTClaimsSchema`) — per #1012

### vitest 4.1.0 Utilities to Leverage (from #1089)

- `vi.fn().mockThrow(err)` — cleaner than `mockImplementation(() => { throw err })`
- `vi.doMock()` disposable — `using mock = vi.doMock(...)` auto-cleanup pattern
- `vi.setTickMode()` — fine timer control for `withRetry` delay testing
- `MatcherResult` / `CustomMatcher` types — typed custom matchers for AGTree AST assertions

## 3. Test Infrastructure Consolidation

### Recommended Layered Stack

| Layer | Framework | Scope |
|---|---|---|
| `src/` pure unit tests | Deno native (`Deno.test` + `@std/assert`) | Pure functions, Zod schemas, platform abstractions |
| `worker/` binding tests | `@cloudflare/vitest-pool-workers` | KV/D1/R2/Queue/Browser binding behavior |
| E2E / UI tests | `@cloudflare/playwright` (BROWSER binding) | Angular frontend, compile flow, visual regression |
| AGTree roundtrip | Deno parameterized tests | Format-pair fidelity (A→B→A) per #993 |

### Create `tests/fixtures/` Shared Module

```
tests/
  fixtures/
    sample-rules/
      hosts.txt
      adblock.txt
      rpz.txt
    mocks/
      MockBrowserFetcher.ts     # IPlaywrightBrowser structural mock
      MockContentFetcher.ts     # IContentFetcher mock
      MockEnv.ts                # Minimal Cloudflare Env mock
    factories/
      compiler-config.ts        # Factory: WorkerCompilerOptions test defaults
      compilation-config.ts     # Factory: CompilerConfiguration with test sources
```

### AGTree Roundtrip Fidelity Tests (Related: #993)

```typescript
const formatPairs = [["adguard", "hosts"], ["hosts", "rpz"], ["ublock", "adguard"]];

for (const [src, tgt] of formatPairs) {
  Deno.test(`Roundtrip fidelity: ${src} → ${tgt} → ${src}`, () => {
    const original = loadSampleRules(src);
    const translated = translate(original, src, tgt);
    const roundtripped = translate(translated, tgt, src);
    assertFidelityAboveThreshold(original, roundtripped, 0.95);
  });
}
```

### CI Integration (Related: #986)

- [ ] `deno test src/` on every PR (unit tests)
- [ ] `vitest --pool=workers` on every PR (Worker binding tests)
- [ ] E2E Browser Rendering tests on deploy to staging only
- [ ] `deno bench --json > benchmarks.json` — upload as CI artifact on `main` merge
